# AGENTS.md

Guide for AI coding agents working in this repo. Keep changes small, focused, and consistent with the conventions below.

## Project

`chime-cli` — a zero-dependency Python CLI for timers, clock-time alarms, and pomodoro sessions. Targets macOS, Linux, and Windows. Requires Python >=3.10. Uses only the standard library at runtime.

## Layout

- `src/chime/` — package (src-layout). Entry point: `chime.cli:main`.
  - `cli.py` — argparse plumbing and user-facing output. No OS calls.
  - `parsers.py` — pure functions (duration/time parsing, formatting).
  - `alerts.py` — side-effecting OS calls (notify, sound, speech). Dispatches on `sys.platform`.
  - `state.py` — background-alarm persistence (`alarms.json`).
  - `term.py` — ANSI color helpers.
  - `_version.py` — generated by hatch-vcs at build time; never edit.
- `tests/` — pytest suite, mirrors module names.
- `scripts/release.py` — release helper (used via `make release`).
- `docs/` — `CONTEXT.md` (domain language), ADRs, PRDs, roadmap.

## Module discipline

- Pure logic → `parsers.py`. OS side effects → `alerts.py` or `state.py`. argparse + printing → `cli.py`. Do not blur these boundaries.
- Cross-platform support in `alerts.py` is keyed on `sys.platform`. When adding a backend, extend `notify()`, `play_sound()`, `speak()`, and `list_sounds()` together.
- Keep the runtime dependency list empty. Anything new must come from the stdlib.

## Dev commands

Use `make` — it is the canonical entry point.

- `make install` — editable install with dev extras + pre-commit hooks.
- `make test` — pytest.
- `make test-cov` — coverage report.
- `make lint` — `ruff check` + `ruff format --check`.
- `make fmt` — auto-fix and reformat.
- `make build` — sdist + wheel.
- `make smoke` — end-to-end check against the installed `chime` command.
- `make help` — list all targets.

Always run `make lint && make test` before reporting work complete.

## Code style

- Ruff is the only formatter/linter; config in `pyproject.toml` (`line-length = 100`, `target-version = "py310"`).
- `from __future__ import annotations` at the top of every module.
- Type hints encouraged; no mypy gate.
- Double-quoted strings (ruff `quote-style = "double"`).
- Match existing patterns before introducing new ones. No new abstractions unless the task requires them.

## Tests

- pytest with `filterwarnings = ["error"]` — any warning fails the suite.
- `--strict-markers` and `--strict-config` are on; do not add markers without registering them.
- Add a test for every behavior change.
- Tests for `parsers.py` should stay pure; for `alerts.py`/`state.py`, mock the OS surface — do not actually play sounds or write to the real state dir.

## Domain language

When touching anything alarm- or timezone-related, read `docs/CONTEXT.md` first and use the canonical terms (Timer, Alarm, Pomodoro session, Background alarm, Configured/System/Source/Effective timezone, Zone alias). Avoid the listed informal alternates.

## Versioning & releases

- Version is derived from the git tag by hatch-vcs. Never hand-edit `_version.py` or add a version string.
- Releases go through `scripts/release.py` (`make release[-minor|-major|-dry]`). Do not tag, publish, or push to PyPI from an agent session unless explicitly asked.
- Update `CHANGELOG.md` under `[Unreleased]` for any user-visible change.

## Things to avoid

- Adding runtime dependencies.
- Bypassing pre-commit (`--no-verify`) or skipping CI checks.
- Editing `_version.py`, `CHANGELOG.md` released sections, or git tags without explicit instruction.
- Wide refactors bundled with a bug fix or feature — keep PRs scoped.
- Touching `src/chime/__pycache__/`, `.venv/`, or other generated/local-only paths.
